# PostCAD Pilot Evidence Bundle

The evidence bundle captures a full local pilot run as a set of inspectable files.
It is generated by `release/generate_evidence_bundle.sh` and written to `release/evidence/current/`.

---

## Generate it

Requires the service to be running (start with `./release/start_pilot.sh`):

```bash
./release/generate_evidence_bundle.sh
```

Replaces any existing `release/evidence/current/` cleanly each time.

---

## Folder structure

```
release/evidence/current/
├── summary.txt                   human-readable summary of what was captured
├── commands.txt                  list of HTTP calls and fixture paths used
├── git_head.txt                  git commit hash at time of capture
│
├── inputs/
│   ├── case.json                 pilot case input (canonical fixture)
│   ├── registry_snapshot.json   manufacturer registry used for routing
│   └── config.json              routing config (jurisdiction + policy)
│
├── 01_health.json                GET /health response
├── 02_store_case.json            POST /cases response
├── 03_route_case.json            POST /cases/:id/route response (receipt + derived_policy)
├── 04_receipt.json               GET /receipts/:hash response
├── 05_dispatch.json              POST /dispatch/:hash response (or 409 note if re-run)
├── 06_verify.json                POST /dispatch/:hash/verify response
├── 07_route_history.json         GET /routes response
│
└── data_artifacts/
    ├── cases/                    files written by the service under data/cases/
    ├── receipts/                 files written by the service under data/receipts/
    ├── policies/                 files written by the service under data/policies/
    ├── dispatch/                 files written by the service under data/dispatch/
    └── verification/             files written by the service under data/verification/
```

`data_artifacts/` is present only if the service wrote to the default `data/` path.

---

## What to inspect first

1. **`summary.txt`** — confirms all 7 steps passed, shows `outcome=routed` and `result=VERIFIED`
2. **`03_route_case.json`** — the full routing receipt with all hash commitments
3. **`06_verify.json`** — verification result must be `{"result":"VERIFIED"}`
4. **`04_receipt.json`** — the receipt as stored by the service (must match step 3)
5. **`inputs/`** — the exact fixture inputs used; a reviewer can re-run verification independently

---

## What "good" looks like

`summary.txt` ends with:

```
outcome   = routed   (expected: routed)
result    = VERIFIED (expected: VERIFIED)
routes    = 1 route(s) in history

All 7 steps passed.
```

`06_verify.json` contains:

```json
{
  "receipt_hash": "<hash>",
  "result": "VERIFIED"
}
```

---

## Notes

- `release/evidence/current/` is excluded from git (listed in `.gitignore`).
- Re-running the generator always replaces `current/` with a fresh capture.
- No timestamps are used in file names; the bundle is deterministic across runs.
- The pilot case_id is always `f1000001-0000-0000-0000-000000000001`.
